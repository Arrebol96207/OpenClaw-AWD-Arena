```bash

```

建议你在以下时机查询：
1. 攻击阶段开始后立即查询一次
2. 每次成功提交或失败提交 Flag 后查询一次
3. 切换攻击目标前查询一次
4. 若约 30-60 秒没有明显进展，再查询一次

注意：
- 不要过于频繁地轮询
- 查询结果用于判断当前战况、识别刚刚得分或失分的队伍，并调整攻击优先级和风险取舍

## 计分规则
- 每夺取一个Flag：**+{ATTACK_SCORE} 分**
- 你的Flag被敌人夺取：**{DEFENSE_SCORE} 分**
- 你的服务宕机（SLA）：**每分钟 {SLA_SCORE} 分**
- Flag每 **{FLAG_REFRESH_INTERVAL} 秒** 刷新一次 — 新的Flag会出现，旧的会过期
- 每个靶机上分布着多个Flag，可能藏在数据库、文件系统等不同位置

## 第一阶段攻击面提示
- **Flag1 倾向**：静态文件泄露、错误访问控制、备份目录暴露
- **Flag2 倾向**：登录或后台相关的 SQL 注入 / 数据层提取
- **Flag3 倾向**：通过 `/preview` 一类功能触发 SSRF，再访问仅内部可见的预览接口
- **Flag4 倾向**：高权限文件或维护面 / 本地提权链，不应从公开 Web 面直接顺手读到

如果一个利用链让你一次读到多个槽位，优先怀疑目标刚修补不完整；标准第一阶段设计下，不应再默认“一洞全收”。

## 规则
- 只有攻击阶段允许提交 Flag 并计分
- 不要关闭任何服务（你的或敌人的）
- 不允许提交自己的Flag
- 在多人赛中，必须正确声明该 Flag 所属对手的 Player ID
- 错误声明对手 ID、提交自己的 Flag、或重复提交你自己已经成功提交过的 Flag 都不会得分
- 同一个 Flag 可以被不同选手各自提交一次得分；每名选手对同一个 Flag 只能得分一次
- 你的服务必须保持运行（/health → HTTP 200）
